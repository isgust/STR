Analogias e o Conceito de Tempo Real

Neste projeto, modelamos situações do mundo real através da mecânica do jogo. Abaixo vemos como cada peça se encaixa na teoria de Sistemas de Tempo Real:

💣 A Bomba = O Deadline

A explosão atua como o Deadline Absoluto. As tarefas devem terminar antes desse limite. Perder esse prazo gera um Deadline Miss, causando a derrota do jogador.

🚦 Soft Real-Time

Este sistema é classificado como Tempo Real Frouxo (Soft). A falha no deadline não causa falha crítica ou danos catastróficos ao sistema (ao contrário do freio de um carro), gerando apenas uma degradação de serviço aceitável.

🏃 Os 3 Jogadores = Tarefas (Tasks)

Os jogadores representam Tasks concorrentes com atrasos de rede variáveis:

  • P1 (20ms): Uma Task rápida, com tempo de comunicação previsível.
  • P2 (150ms) e P3 (400ms): Tasks que sofrem com jitter e atrasos de propagação severos, provando a necessidade de tolerância a falhas temporais e compensação por parte do servidor.

Concorrência e Tasks (Tarefas)

Onde a concorrência acontece de verdade no sistema? Em jogos multiplayer, conexões simultâneas precisam ser tratadas juntas.

No Servidor (Go): O modelo de Goroutines é utilizado extensivamente. Em vez de bloquear o sistema para esperar a mensagem de um cliente, dividimos o trabalho em várias Tasks Assíncronas:

  • readPump: Task de Leitura (Event-driven). Fica esperando entradas de rede a todo instante, lendo os comandos dos jogadores (ações de pressionar).
  • writePump: Task de Escrita. Seu único trabalho é despachar mensagens para a rede de volta aos clientes via Channels.
  • Game.Run: Task de Broadcast / Loop do Servidor. Orquestra a central de eventos do jogo sem causar bloqueios (I/O).
  • Timer Task: Uma tarefa de interrupção (time.AfterFunc) que executa a "explosão" exatamente após decorrido o Delay.

Para evitar corrupção de memória quando múltiplos jogadores enviam a resposta simultaneamente (Race Conditions), protegemos o estado da partida com Exclusão Mútua através de sync.Mutex (g.mu.Lock / Unlock).

Algoritmo de Escalonamento: O Go Scheduler

Como essas Tasks recebem tempo de processamento da CPU? Utilizamos o poder do próprio Runtime do Golang.

O Go implementa um escalonador embutido M:N Work-Stealing. Ele mapeia logicamente M goroutines em N threads do Sistema Operacional.

O Modelo G-P-M

  • G (Goroutine): A tarefa / task em execução.
  • M (Machine): Thread física do Sistema Operacional rodando na CPU.
  • P (Processor): Processador lógico do Go. Ele mantém uma Fila Local de Goroutines prontas.

Preempção Cooperativa por I/O

No nosso jogo, readPump fica esperando na rede (I/O bloqueante). Quando a Thread do SO (M) trava esperando os dados de I/O, o Escalonador age rapidamente: ele desvincula o P dessa thread travada e entrega o P para uma nova Thread M livre. Assim, as outras Goroutines na fila daquele P continuam rodando fluidamente!

Work-Stealing (Roubo de Trabalho)

Se o processador P1 zerar sua fila local enquanto P2 está sobrecarregado, o algoritmo de Work-Stealing entra em ação: P1 "rouba" metade do trabalho da fila de P2 para si, balanceando a carga em tempo real.

Lag Compensation (Compensação de Atraso)

Se o servidor não compensasse a latência, o P3 (com 400ms) sempre perderia o prazo mesmo agindo na hora certa na tela dele. A compensação resolve a "injustiça do Deadline".

No código abaixo (em game.go), o servidor "volta no tempo" deduzindo o atraso:

func (g *Game) handlePress(player int, simulatedLatency int64) {
    // Atraso de simulação na rede
    time.Sleep(time.Duration(simulatedLatency) * time.Millisecond)

    serverReceiveTime := time.Now().UnixMilli()
    var eventTime int64

    if g.lagComp {
        // VOLTA NO TEMPO: Subtrai a latência (Ping) do momento em 
        // que o pacote chegou. Encontra a hora exata da ação real!
        eventTime = serverReceiveTime - simulatedLatency
    } else {
        // Injusto: Assumimos que o cliente apertou agora.
        eventTime = serverReceiveTime
    }

    // Avalia se cumpriu o Deadline (Prazo da explosão)
    if eventTime >= g.explosionTime {
        status = "DEAD"  // Deadline Miss!
    } else {
        status = "ALIVE" // Deadline Cumprido!
    }
}
